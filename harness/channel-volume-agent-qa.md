# 채널물량 인사이트 AI AGENT 예상 질문 답변집

작성일: 2026-06-12  
용도: 기획 검토, 임원 보고, 유관부서 질의응답 대응

---

## 질문 1. 가장 약한 가정은 무엇인가요?

### 짧게 답하면

가장 약한 가정은 `월별 시책과 이슈 데이터를 충분히 구조화해서, 총계 기준의 숫자 예측과 채널별 설명을 동시에 안정적으로 만들 수 있느냐`입니다.

### 조금 더 설명하면

이 AGENT의 난점은 숫자 데이터가 없는 것이 아니라, `숫자`와 `맥락`이 분리되어 있다는 점입니다.  
업적 물량, 조직 효율, 시장 규모 같은 숫자는 비교적 정리돼 있지만, 시책 세부내용, 신상품 영향, 채널별 현장 이슈는 문서와 사람 기억에 흩어져 있습니다.

그래서 가장 중요한 전제는 다음과 같습니다.

1. 월별/주별/일별 물량 데이터가 충분히 정리돼 있어야 한다.
2. 월별 시책과 이슈가 최소한 재사용 가능한 형태로 입력돼야 한다.
3. AI가 시책 효과를 단정이 아니라 `다른 변수와 함께 비교 해석`할 수 있어야 한다.

즉, 모델 성능보다 먼저 `입력 데이터 구조화`가 가장 큰 리스크입니다.

### 방어적으로 답하면

그래서 저희도 처음부터 전사 전체를 한 번에 하려는 것이 아니라,  
`작은 범위의 파일럿`으로 시작해서 데이터 구조화 가능성과 예측 설명력을 먼저 검증하려고 합니다.

## 질문 2. 데이터는 어떻게 할 건가요?

### 짧게 답하면

처음부터 완전 자동 수집으로 가지 않고,  
`정형 숫자 데이터 + 운영자가 넣는 월별 시책/이슈 데이터`를 결합하는 방식으로 시작할 계획입니다.

### 상세 답변

데이터는 크게 3층으로 보려고 합니다.

### 1층. 정형 숫자 데이터

- 월별/주별/일별 업적 물량
- 채널별 물량
- 조직 효율 숫자
- 영업일수
- CSM
- 시장 규모

이 층은 예측의 기본 골격이 됩니다.

### 2층. 반정형 운영 데이터

- 월별 시책 세부내용
- 신상품 관련 정보
- 월간/주간 보고서
- 채널별 특이사항

이 층은 숫자의 해석과 설명 근거를 강화합니다.

### 3층. 운영 입력

- 담당자가 직접 입력하는 핵심 이슈
- 기존 회의자료/보고서 업로드

이 층은 완전 자동화보다 현실적인 운영성을 확보하는 역할을 합니다.

### 운영 원칙

1. 처음부터 메일/메신저까지 전부 자동 수집하지는 않는다.
2. 시책 강도는 사람이 억지로 숫자화하지 않는다.
3. 대신 AI가 과거 패턴, 영업일수, 시장규모, 최근 흐름과 함께 시책 효과를 해석한다.
4. 최근 1~2년 데이터는 더 강하게 보고, 변수에 따라 장기 패턴은 보조적으로 반영한다.

### 한 줄로 정리하면

`정형 숫자는 시스템으로, 월별 맥락은 운영 입력으로, 해석은 AI로` 가져가는 구조입니다.

## 질문 3. 보안 위협은 없나요?

### 짧게 답하면

보안 위협은 있습니다.  
다만 이 프로젝트는 처음부터 `민감한 내부 영업 데이터`를 다루는 전제로 설계해야 하고, 그래서 권한, 반출, 로그, 승인 통제를 같이 넣어야 합니다.

### 예상 보안 위협

1. 내부 실적 데이터와 전략 정보의 외부 반출 위험
2. 권한이 없는 사용자가 다른 채널 데이터까지 조회하는 위험
3. 업로드 문서에 포함된 민감정보가 그대로 모델 입력으로 들어가는 위험
4. 로그나 저장소에 민감한 숫자와 문장이 장기간 남는 위험
5. 보고용 문장이 조직 밖으로 잘못 공유되는 위험

### 대응 방향

1. 권한 통제
   - 사용자별 조회 범위를 채널/조직 기준으로 제한
2. 데이터 최소화
   - 필요한 데이터만 넣고, 불필요한 식별정보는 제외
3. 업로드 통제
   - 허용 파일 형식과 업로드 주체를 제한
4. 감사 로그
   - 누가 어떤 데이터를 조회했고 어떤 예측을 봤는지 기록
5. 사람 승인
   - 공식 보고 전에는 사람 검토를 거치게 함
6. 외부 전송 최소화
   - 어떤 데이터가 외부 AI 호출에 포함되는지 명확히 관리

### 보안 질문이 더 들어오면

이 AGENT는 단순 편의 기능이 아니라 내부 영업 인사이트 시스템이기 때문에,  
`기능 설계`와 `보안 설계`를 같이 묶어서 가야 한다고 답하는 것이 안전합니다.

## 질문 4. 비용은 얼마 정도 나오나요?

### 짧게 답하면

정확한 비용은 `사용자 수`, `질의량`, `문서 적재 빈도`, `모델 선택`에 따라 달라집니다.  
다만 파일럿은 소수 사용자 기준으로 시작해서 월 비용을 통제 가능한 수준에서 검증하는 방식으로 가는 것이 맞습니다.

### 답변 프레임

비용은 크게 3가지로 나뉩니다.

1. 데이터 적재/전처리 비용
2. 저장/검색 비용
3. AI 질의응답 및 예측 비용

### 설명 포인트

- 처음에는 전사 배포가 아니라 `파일럿`이기 때문에 비용 상한을 걸고 검증하는 게 맞다
- 실무적으로는 `모든 데이터를 매번 다시 읽는 구조`보다 `요약/검색/캐시 구조`를 넣으면 비용을 많이 줄일 수 있다
- 조회형 기능은 상대적으로 저렴하고, 예측형/설명형이 비용을 더 쓴다

### 보수적인 답변 예시

`현재 단계에서는 정확한 월 운영비를 단정하기보다, 파일럿 범위 기준으로 사용량을 제한하고 비용 상한을 정해 검증하는 방식이 적절합니다. 핵심은 처음부터 큰 비용을 쓰는 것이 아니라, 작은 사용자군에서 조회 빈도, 문서 적재량, 예측 사용량을 보면서 운영비 곡선을 확인하는 것입니다.`

### 숫자를 꼭 묻는 경우

`지금은 사용량 전제가 없어서 확정 숫자를 말씀드리긴 어렵고, 파일럿 범위를 정하면 월별 데이터 적재량, 질의 수, 사용자 수를 기준으로 별도 시뮬레이션해서 제시드리겠습니다.`

이 답변이 가장 안전합니다.  
지금 단계에서 근거 없이 비용 숫자를 단정하는 것이 오히려 더 위험합니다.

## 질문 5. 사람은 언제 개입하나요?

### 짧게 답하면

사람은 `공식 보고 전`, `예측이 크게 변했을 때`, `근거가 약할 때`, `민감한 해석이 포함될 때` 반드시 개입합니다.

### 상세 답변

이 AGENT는 사람을 대체하는 구조가 아니라,  
`사람이 더 빠르게 보고하고 판단하게 돕는 보조 에이전트`입니다.

그래서 사람 개입 시점은 명확해야 합니다.

### 개입 시점 A. 공식 보고 전

- 임원 보고
- 채널장 공유
- 공식 문서 반영

이 단계에서는 사람이 숫자, 근거, 문장을 최종 검토해야 합니다.

### 개입 시점 B. 예측 숫자가 크게 변했을 때

- 전주 대비 전망이 급변한 경우
- 월중 갱신 결과가 크게 달라진 경우

이 경우 사람은 원인과 입력 누락 여부를 다시 확인해야 합니다.

### 개입 시점 C. 근거가 약하거나 충돌할 때

- 시책 효과와 시장 변수 해석이 충돌하는 경우
- 최신 이슈 입력이 불완전한 경우
- 채널별 설명이 납득되지 않는 경우

이 경우 AI 답변을 그대로 쓰지 않고, 사람 확인을 전제로 해야 합니다.

### 개입 시점 D. 민감한 조직 해석이 포함될 때

- 특정 채널 부진 원인
- 시책 실패 해석
- 조직 운영 이슈 해석

이런 내용은 오해 소지가 있어 사람이 문장 톤과 노출 범위를 조정해야 합니다.

### 한 줄로 정리하면

`AI는 초안을 만들고, 사람은 최종 보고와 책임을 맡는다`가 원칙입니다.

## 발표 때 한 번에 말하는 요약본

`이 AGENT의 가장 약한 가정은 숫자와 함께 시책/이슈 맥락을 얼마나 구조화할 수 있느냐입니다. 그래서 처음부터 전부 자동화하지 않고, 정형 숫자 데이터와 운영자가 입력하는 월별 시책/이슈를 결합하는 방식으로 파일럿을 시작하려고 합니다. 보안 측면에서는 내부 영업 데이터이기 때문에 권한 통제, 로그, 승인 체계를 같이 설계해야 하고, 비용은 사용자 수와 질의량에 따라 달라지므로 파일럿 기준으로 상한을 두고 검증하는 게 맞습니다. 사람은 공식 보고 전, 예측 급변 시, 근거가 약할 때 반드시 개입하고, 이 AGENT는 최종 판단자가 아니라 사람의 정리·보고를 돕는 보조 도구로 설계하는 것이 핵심입니다.`
